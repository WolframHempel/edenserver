Why EdenServer?

EdenServer continues where AMD left off. It recursively bundles files, resolves dependencies, adds assets such as html and css and runs pipeline tasks such as minification and compilation (e.g.less).

And it does all of that highly efficient while you develop as well as in production.

Starting the server

Fork or download EdenServer from Github. It is written in node.js, so if you haven't installed node yet, you can download it at nodejs.org To start it, run:

						cd server
						node start.js

Overview

This is just a quick introduction into some of the features of EdenServer.

A proper tutorial and documentation will follow soon. For the time being please have a look at the example applications in the /apps directory.

Modules

A module can be any combination of JavaScript, HTML, CSS and other assets. It can depend on other modules. EdenServer will create a closure around your modules so that everything within it is private to that module only.

EdenServer doesnâ€™t expect any specific folder structure. Each module shoud have at least a js OR html OR css folder. Subfolders within these will be scanned and bundled recoursively.

Requiring a module

In order to require a module use eden.require. This function works very much like the require.js equivalent. To require the module from the folder structure example use 

							eden.require( 
								"myApp/modules/stockchart", 
								function( StockChart ){
								 	var chart = new StockChart();
								}
							);

Module specific keywords

Within each module Eden will inject two specific keywords, module and exports. module is used to access assets that are specific to that module, e.g. module.get( "html", "header" ) would return the contents of the header.html file in the modules html folder. module.path contains the path the module had been requested with.

eden.json files

eden.json files are used to configure dependencies and pipeline tasks. They can be placed at any level of the folder structure. They extend each other downwards (a setting in myApp/modules/eden.json would be overwritten by the same setting in myApp/modules/myModule/eden.json) 

					{
					  "dependencies": {
					    "ko": "../thirdparty/knockout",
					    "$": "tplTest/thirdparty/jQuery"
					  }
					}

javascript client

The javascript client is served at /eden.js. It is specific to the instance of the server, so it can not be served from the file system.

http server

EdenServer is also a great http server for static files. It's using node-static internally which handles caching, mimetype lookups and all other http functionality.

What's happening internally?

Whenever a module is requested for the first time eden server reads all files, runs all pipeline tasks and stores a ready-to-be-served string for the entire module content in memory.

It also add's file watcher to every file and folder within that module. If a change occurs, only that file or folder will be re-read and bundled, making EdenServer highly efficient for development.

Oh, right...if a request is made for a module that's still being bundled the request is held and will only be served once the bundling finished.
